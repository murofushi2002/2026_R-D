# 研究設計候補：口コミ活用型価格決定支援エージェントの構築

---

## 前提：現在の研究資産

| 要素 | 現状 |
|------|------|
| ABSA パイプライン | LLM → 観点別理由文抽出 → BERT → 感情スコア |
| 需要予測モデル | 構造化データ + ABSA感情スコア（清掃等）→ 予測精度向上を確認済 |
| 対象ドメイン | 地方宿泊施設（日本） |
| 確認済み知見 | 清掃スコアの追加で需要予測精度が向上する |

以下、この資産を活かして「価格決定支援」に拡張する4つの研究設計候補を提案する。各候補について、**全体アーキテクチャ**、**ABSAの必要な拡張**、**価格設定モデル**、**評価方法**、**文献的根拠**、**強みと弱み**を記述する。

# 候補A：LLM-as-Pricing-Advisor — 対話型価格根拠提示エージェント

## A-1. コンセプト

LLMエージェントが「価格アドバイザー」として機能し、ABSAの分析結果と需要予測を統合して、**自然言語による価格提案と根拠説明**を施設マネージャーに提示する。マネージャーとの対話を通じて「what-if」シナリオを探索可能。

**核心的問い**: ABSAで得た観点別口コミ感情と需要予測結果を、LLMが自然言語で「価格変更の根拠」として説明できるか？ その説明は施設マネージャーの意思決定を改善するか？

## A-2. アーキテクチャ

```
┌─────────────────────────────────────────────────────┐
│  入力層                                               │
│  ・口コミテキスト（じゃらん、楽天トラベル等）              │
│  ・予約データ（稼働率、ADR）                             │
│  ・カレンダー情報（祝日、地域イベント）                    │
│  ・競合施設の価格・評価（任意）                           │
└───────────┬─────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────┐
│  ABSA モジュール（拡張版）                              │
│  LLM → 観点別理由文抽出 + 時系列トレンド集約              │
│  BERT → 感情スコア（5段階: 強ポジ/ポジ/中立/ネガ/強ネガ）  │
│  出力: aspect_scores[t], trend[t-k:t], key_phrases    │
└───────────┬─────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────┐
│  需要予測モジュール（既存拡張）                          │
│  入力: 構造化データ + aspect_scores[t]                  │
│  モデル: GRU / XGBoost / LightGBM                     │
│  出力: predicted_demand[t+1], feature_importance (SHAP)│
└───────────┬─────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────┐
│  LLM 価格アドバイザーエージェント                        │
│  入力:                                                │
│    ・aspect_scores + trend + key_phrases              │
│    ・predicted_demand + SHAP値                         │
│    ・現在価格、競合価格、稼働率                          │
│    ・施設プロフィール（ナレッジベース）                    │
│  処理:                                                │
│    ・Structured Prompt → 価格提案 + 根拠文生成           │
│    ・What-if シナリオ応答                               │
│  出力: 「推薦価格: ¥XX,XXX（根拠: ...）」               │
└───────────┬─────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────┐
│  マネージャー（Human-in-the-loop）                      │
│  ・提案を確認 → 承認 / 修正 / 却下                       │
│  ・対話: 「もし朝食を改善したら？」→ what-if応答          │
└─────────────────────────────────────────────────────┘
```

## A-3. ABSAの拡張

| 拡張項目 | 内容 | 必要性 |
|---------|------|-------|
| **感情スケール拡張** | 3段階（ポジ/中立/ネガ） → 5段階（強ポジ/ポジ/中立/ネガ/強ネガ） | 価格調整の「程度」を判断するため。Arctic-ABSA（Liskowski & Jankowski, 2026）は5クラス+mixed+unknown分類でSoTAを達成 |
| **時系列集約** | 各aspectの感情スコアをウィンドウ集約（週次/月次トレンド） | 「先月比で清掃が悪化」等のトレンド情報が価格根拠に必要 |
| **キーフレーズ抽出の強化** | 理由文から「具体的改善候補」を抽出（例:「浴室のカビ」「朝食の品数」） | LLMの根拠説明に具体的文言が必要。Jeong & Lee (2024, 被引用43件) のChatGPTによるaspect-based summarizationの手法を参考 |
| **（任意）競合比較ABSA** | 競合施設レビューにも同じABSAを適用し、aspect別ポジショニングを得る | 「競合より清掃評価が低い」等の比較根拠。実装コスト高のためオプション |

## A-4. 価格設定モデル

**ルールベース + LLM推論のハイブリッド**

1. **ベースライン価格**: 需要予測 + 過去実績から算出（回帰 or 最適化）
2. **感情調整係数**: aspect感情スコアの変動に応じた価格補正
   - $P_{adjusted} = P_{base} \times (1 + \sum_{a \in A} w_a \cdot \Delta s_a)$
   - ここで $w_a$ はaspect $a$ の価格弾力性（学習）、$\Delta s_a$ は感情スコアの基準期からの変化
3. **LLM推論**: 調整後価格と全情報をプロンプトに入力し、自然言語で根拠と推薦を生成

**LLMの役割**: 最適化の「計算」ではなく、計算結果の「解釈・説明・対話」に特化。Liu et al. (2025, EMNLP) はLLMが複雑な価格計算で系統的に失敗することを示しており、価格の「算出」はMLモデルに委ね、LLMは「根拠の言語化」に限定するのが安全。

## A-5. 評価方法

| 評価軸 | 方法 | 指標 |
|-------|------|------|
| 予測精度 | ABSA拡張版 vs 現行版のablation | RMSE, MAE, R² |
| 価格提案品質 | シミュレーション環境でのRevPAR比較（提案従守群 vs ベースライン） | RevPAR, 稼働率, ADR |
| 説明品質 | 生成された根拠文に対する人間評価 | 理解度、有用性、信頼向上度（5段階Likert） |
| 受容性 | 施設マネージャーへのユーザースタディ（N≥10） | 提案受容率、SUS (System Usability Scale)、自由記述 |
| What-if有効性 | シナリオ質問に対する応答の妥当性を専門家が評価 | 正確性、一貫性 |

## A-6. 文献的根拠

| 設計要素 | 根拠文献 |
|---------|---------|
| HITL設計の必要性 | Ivanov & Webster (2024): マネージャーは価格決定の制御維持を望む |
| XAI説明ニーズ（what, why, what-if） | Mohammed & Denizci Guillet (2025b): 7種の質問駆動型説明 |

| 5段階感情分類 | Liskowski & Jankowski (2026): Arctic-ABSA, CoTによるreasoning injection |
| 口コミ→需要予測の有効性 | Wu et al. (2022, 被引用78件); Degife & Lin (2024) |
| 口コミ→収益への影響 | Abrate, Nicolau, & Viglia (2019, 被引用215件) |

## A-7. 強み・弱み

| | 内容 |
|--|------|
| **強み** | (1) 実装難度が相対的に低い（既存パイプライン+LLMプロンプト設計）、(2) What-if対話機能で実務者のニーズに直接対応、(3) 各モジュールが独立に評価可能、(4) 文献的根拠が厚い |
| **弱み** | (1) LLMのハルシネーション管理が必要、(2) 価格最適化の理論的保証がない（ヒューリスティック的）、(3) LLM推論のレイテンシ |
| **新規性** | 口コミABSA結果を「XAIの説明材料」として使う発想は文献上存在しない（ギャップ2: XAIとレビュー分析の統合） |

# 候補B：マルチエージェント協調型価格決定支援システム

## B-1. コンセプト

**4つの専門エージェント**（口コミ分析・需要予測・価格最適化・説明生成）がLLMベースのオーケストレーターを通じて協調し、各専門知識を統合した価格提案を生成する。

**核心的問い**: マルチエージェント構成は、単一モデルと比較して、価格決定支援の質（精度・説明性・柔軟性）を向上させるか？

## B-2. アーキテクチャ

```
                    ┌────────────────────────────┐
                    │  Orchestrator Agent (LLM)   │
                    │  タスク分解・エージェント間調整  │
                    │  最終的な提案文生成            │
                    └──┬─────┬─────┬─────┬───────┘
                       │     │     │     │
         ┌─────────────┘     │     │     └──────────────┐
         ▼                   ▼     ▼                    ▼
┌────────────────┐ ┌──────────────┐ ┌──────────────┐ ┌────────────────┐
│ Review Analyst │ │  Demand      │ │  Price       │ │  Explanation   │
│ Agent          │ │  Forecaster  │ │  Optimizer   │ │  Generator     │
│                │ │  Agent       │ │  Agent       │ │  Agent         │
│ ・ABSA実行     │ │ ・需要予測   │ │ ・価格算出   │ │ ・根拠文生成   │
│ ・トレンド分析 │ │ ・特徴量重要度│ │ ・制約最適化 │ │ ・What-if応答  │
│ ・改善提案抽出 │ │ ・不確実性推定│ │ ・感度分析   │ │ ・可視化指示   │
└────────────────┘ └──────────────┘ └──────────────┘ └────────────────┘
```

**各エージェントの詳細**:

### Agent 1: Review Analyst（口コミ分析エージェント）
- 拡張ABSAパイプラインを実行
- aspectごとの感情スコア + 時系列トレンド + 代表的原因フレーズを出力
- **拡張**: 「改善可能性スコア」を付与（「立地」は変更不可→低、「清掃」は改善可能→高）
- ツール使用: ABSA model API, レビュー取得API

### Agent 2: Demand Forecaster（需要予測エージェント）
- 既存の需要予測モデルを呼び出し、予測値 + SHAP値 + 予測不確実性を出力
- **拡張**: 予測区間（prediction interval）の推定により「自信度」を提示

### Agent 3: Price Optimizer（価格最適化エージェント）
- 制約付き最適化を実行
- 決定変数: 価格 $p$
- 目的関数: $\max_{p} \; p \cdot D(p, \mathbf{s})$ （収益最大化）
  - $D(p, \mathbf{s})$: 需要関数（Agent 2の予測に基づく）、$\mathbf{s}$: aspect感情ベクトル
- 制約: 価格上下限、常連客割引制約、競合価格範囲

### Agent 4: Explanation Generator（説明生成エージェント）
- Agent 1〜3の全出力を受け取り、自然言語の提案文を生成
- Mohammed & Denizci Guillet (2025b) の7種質問型（what, how, why, why-not, what-if, how-to, what-else）に対応
- 250〜300語の箇条書き形式（同文献の「簡潔さ」要件）

### Orchestrator（統括エージェント）
- LLMベース（LangGraph / CrewAI等で実装）
- マネージャーからの質問を適切なエージェントにルーティング
- エージェント間の結果を統合・矛盾解消

## B-3. ABSAの拡張

| 拡張項目 | 内容 | 候補Aとの差分 |
|---------|------|-------------|
| **感情スケール拡張（5段階）** | 候補Aと同じ | 同一 |
| **時系列集約** | 候補Aと同じ | 同一 |
| **改善可能性スコア** | 各aspectに「施設が改善可能か」の二値ラベル | B固有。「改善できるaspectの悪化」と「改善不可aspectの悪化」で価格提案を変える |
| **Aspect Sentiment Quad Prediction** | (aspect term, aspect category, sentiment, opinion term) の4つ組を同時予測 | B固有。Ventirozos et al. (2025, RANLP) のASQP手法を参考。より構造化された出力 |
| **Chain-of-Thought ABSA** | BERTの感情分類にCoT的推論チェーンを付与 | B固有。Liskowski & Jankowski (2026) のreasoning pretraining technique |

## B-4. 価格設定モデル

**Agent 3における制約付き最適化**:

$$\max_{p} \; r(p) = p \cdot \hat{D}(p, \mathbf{s}, \mathbf{x})$$

ここで:
- $\hat{D}$: Agent 2の需要予測モデル
- $\mathbf{s} = (s_1, ..., s_K)$: $K$ 個のaspect感情スコア
- $\mathbf{x}$: 外部特徴量（曜日、季節、イベント、競合価格）

制約条件:
- $p_{min} \leq p \leq p_{max}$（施設設定の価格レンジ）
- $|p - p_{prev}| \leq \delta$（急激な価格変動の抑制；常連客配慮）
- $p \leq f(p_{comp})$（競合価格に対する上限設定；任意）

**需要関数の推定**: Agent 2が出力する予測値を複数価格点で評価し（Agent 3が価格シナリオを生成）、需要曲線を近似的に構築。

## B-5. 評価方法

| 評価軸 | 方法 | 指標 |
|-------|------|------|
| **エージェント協調** | Ablation: 4エージェント完全版 vs 各エージェント除去版 | 提案品質（人間評価）、収益シミュレーション |
| **価格最適化** | シミュレーション環境での収益比較 | RevPAR, 稼働率, 収益 |
| **ASQP精度** | SemEval14 + ホスピタリティドメインデータでの評価 | F1 (aspect extraction, sentiment classification) |
| **受容性** | ユーザースタディ | SUS, TAM (Technology Acceptance Model), 受容率 |
| **説明品質** | 7種質問型への応答を専門家評価 | 正確性、網羅性、有用性 |

## B-6. 文献的根拠

| 設計要素 | 根拠文献 |
|---------|---------|
| マルチエージェントの宿泊分野応用 | Dwivedi et al. (2026, IJCHM, 被引用25件): ホスピタリティにおけるAIエージェントの機会と課題 |

## B-7. 強み・弱み

| | 内容 |
|--|------|
| **強み** | (1) 各エージェントが独立にテスト・改良可能、(2) 最も体系的な設計で論文としての貢献が大きい、(3) 最適化の理論的根拠がある、(4) 「AIエージェント in hospitality」が2026年のホットトピック |
| **弱み** | (1) 実装複雑度が最も高い、(2) エージェント間通信のオーバーヘッド、(3) 評価に時間がかかる、(4) 修士論文のスコープとしてはやや大きい可能性 |
| **新規性** | 宿泊産業における口コミ駆動型マルチエージェント価格支援は先例なし |

# 候補C：ABSA強化型強化学習（RL）+ LLM事後説明

## C-1. コンセプト

ABSAで得た感情スコアをRLエージェントの**状態表現**に組み込み、価格調整方策を学習させる。学習済み方策の各価格決定に対してLLMが事後的に自然言語の説明を生成する。

**核心的問い**: aspect感情スコアを状態変数として含むRL方策は、含まない方策と比較して、累積収益を有意に改善するか？

## C-2. アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│  環境 (Environment)                                      │
│  シミュレーション: 需要モデル D(p, s, x) + 口コミ生成モデル  │
│                                                          │
│  状態 s_t = [occupancy_t, dow_t, season_t, event_t,      │
│              aspect_sentiment_vector_t, competitor_p_t,   │
│              review_volume_t, price_t-1]                  │
│                                                          │
│  行動 a_t = Δp ∈ {-15%, -10%, -5%, 0%, +5%, +10%, +15%} │
│                                                          │
│  報酬 r_t = α·revenue_t + β·occupancy_t - γ·|Δp_t|      │
│            （収益 + 稼働率ボーナス - 価格変動ペナルティ）    │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  RLエージェント                                           │
│  アルゴリズム: DQN / PPO / SAC                            │
│  Q-network入力: 状態ベクトル (含 aspect感情スコア)         │
│  出力: 離散的価格調整行動                                  │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  事後説明モジュール                                        │
│  (1) SHAP: Q値に対する各特徴量の貢献度                     │
│  (2) LLM: SHAP値 + 状態情報 → 自然言語説明                │
│  出力: 「本日+5%を推薦。理由: 清掃スコアが先週比+0.3上昇、 │
│         競合価格が平均+8%、予測稼働率85%」                 │
└─────────────────────────────────────────────────────────┘
```

## C-3. ABSAの拡張

| 拡張項目 | 内容 | 必要性 |
|---------|------|-------|
| **感情スケール拡張（5段階→連続値）** | BERTの出力をsoftmax確率から連続スコア [0,1] に変換 | RLの状態表現として連続値が必要 |
| **時系列集約** | 各aspectの移動平均・移動分散を特徴量化 | 状態空間にトレンド情報を含める |
| **Aspect重要度の自動推定** | 需要予測モデルのSHAP値からaspect重要度を動的に推定 | Degife & Lin (2024) のaspect感度分析を一般化 |
| **レビュー量の特徴量化** | 期間あたりのレビュー件数を状態に含める | レビュー量自体が需要の先行指標 |

## C-4. 価格設定モデル

**深層強化学習によるMDP定式化**:

状態空間:
$$\mathbf{s}_t = (\text{occ}_t, \text{dow}_t, \text{season}_t, \text{event}_t, s^{(1)}_t, \ldots, s^{(K)}_t, p^{comp}_t, n^{rev}_t, p_{t-1})$$

行動空間（離散）:
$$a_t \in \{-15\%, -10\%, -5\%, 0\%, +5\%, +10\%, +15\%\}$$

報酬関数:
$$r_t = \alpha \cdot p_t \cdot d_t + \beta \cdot \mathbb{1}[\text{occ}_t > \theta] - \gamma \cdot |p_t - p_{t-1}|$$

- 第1項: 収益（価格×実需）
- 第2項: 稼働率閾値ボーナス（地方施設では空室回避が優先される場合がある）
- 第3項: 価格変動ペナルティ（常連客配慮、Mohammed & Denizci Guillet 2025a のオーバーライド頻度低減に対応）

**シミュレーション環境の構築**:
- 既存の需要予測モデルを環境の需要関数として利用
- 口コミのダイナミクス: $s^{(k)}_{t+1} = f(s^{(k)}_t, p_t, d_t) + \epsilon$ （価格が高いのにサービスが低いとネガティブ感情が増加する、等の遷移モデル）

## C-5. 評価方法

| 評価軸 | 方法 | 指標 |
|-------|------|------|
| **RL方策性能** | シミュレーション環境での比較（ABSA有 vs 無 vs 固定価格 vs ルールベース） | 累積収益、平均稼働率、学習曲線の収束速度 |

| **受容性** | マネージャーにRL方策の推薦結果を提示し反応を調査 | 提案受容率、信頼度 |

# 候補D：因果推論ベース Aspect-Price Elasticity 推定 + LLM対話

## D-1. コンセプト

ABSAの結果を需要/価格との**因果関係**として推定し、「特定aspectの改善/悪化が、需要と最適価格にどれだけ影響するか」を定量化する。LLMは因果推定結果を「もしXを改善したらYが変わる」というカウンターファクチュアル説明として提示する。

**核心的問い**: 各aspectの感情変化が需要に与える因果的影響（aspect-price elasticity）を識別できるか？ その因果的知見は、「相関的知見」よりも施設マネージャーの信頼と意思決定を改善するか？

## D-2. アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│  ABSA モジュール（拡張版）                                 │
│  → aspect感情スコア s^(k)_t + レビュー量 n^(k)_t          │
└───────────┬─────────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────────┐
│  因果推定モジュール                                        │
│                                                          │
│  (1) Aspect-Demand因果推定                                │
│      ・操作変数法 (IV) / 差分の差分法 (DiD) /               │
│        DoWhy等の因果推論ライブラリ                          │
│      ・内生性問題の対処: 高需要→高評価の逆因果を排除         │
│      ・出力: aspect k の感情1単位変化 → 需要Δd%            │
│                                                          │
│  (2) Aspect-WTP (Willingness to Pay) 推定                 │
│      ・ヘドニック価格モデル: ln(p) = Σ β_k·s^(k) + ...     │
│      ・β_k = aspect k の WTP弾力性                         │
│      ・出力: 清掃スコア+1 → 支払意思額 +¥X                 │
└───────────┬─────────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────────┐
│  カウンターファクチュアルシミュレーター                       │
│  「もし清掃スコアが0.5改善したら？」→ 需要+X%, 最適価格+¥Y  │
│  「もし食事が-0.3悪化したら？」→ WTP-¥Z、稼働率-W%         │
└───────────┬─────────────────────────────────────────────┘
            ▼
┌─────────────────────────────────────────────────────────┐
│  LLM 対話エージェント                                      │
│  因果推定結果をself-containedな説明文に変換                  │
│  「清掃スコアが0.5改善すると需要は+8%増加し、                │
│   ¥1,200の価格引き上げ余地が生じます（因果推定に基づく）」    │
│  対話: マネージャーの「もし〜したら？」に因果モデルで応答    │
└─────────────────────────────────────────────────────────┘
```

## D-3. ABSAの拡張

| 拡張項目 | 内容 | 必要性 |
|---------|------|-------|
| **感情スケール: 連続値** | 候補Cと同じ | 因果推定の説明変数として連続値が必要 |
| **時系列集約** | 全候補共通 | DiDや時系列因果推定に必要 |
| **外生変動の識別用特徴量** | レビュー投稿者の属性（国籍、旅行目的等）をメタデータとして活用 | 操作変数法で内生性に対処する場合に利用 |
| **Aspect Sentiment Quad** | 候補Bと同じ | 因果推定の精度向上（aspect termとopinion termの区別が重要） |
| **施設改善イベントの記録** | 「浴室リフォーム実施日」等の施設側イベントデータ | DiDにおけるtreatment変数 |

## D-4. 価格設定モデル

**ヘドニック価格モデル + カウンターファクチュアル**:

$$\ln(p^*_t) = \alpha + \sum_{k=1}^{K} \beta_k \cdot s^{(k)}_t + \mathbf{\gamma}' \mathbf{x}_t + \epsilon_t$$

- $\beta_k$: aspect $k$ の支払意思額弾力性（=aspect-price elasticity）
- 因果的識別: $s^{(k)}$ の内生性を操作変数等で対処

**カウンターファクチュアル提案**:
$$\Delta p^* = \sum_{k \in K_{changed}} \hat{\beta}_k \cdot \Delta s^{(k)}$$

例: 清掃スコアが $\Delta s^{(clean)} = +0.5$ 改善 → $\Delta p^* = \hat{\beta}_{clean} \times 0.5 = +¥1,200$

## D-5. 評価方法

| 評価軸 | 方法 | 指標 |
|-------|------|------|
| **因果推定の妥当性** | 偽検定 (placebo test)、ロバスト性チェック（異なるIV、異なるモデル仕様） | 係数の有意性、信頼区間の幅 |
| **カウンターファクチュアルの忠実度** | 実際の施設改善イベント前後のデータでback-test | 予測 vs 実績のΔ需要誤差 |
| **説明の因果的質** | 「因果説明あり」vs「相関説明のみ」を施設マネージャーに提示し比較 | 信頼度、意思決定変容率（対照実験） |
| **WTP推定精度** | ヘドニックモデルのR²、交差検証 | R², RMSE |
| **受容性** | 候補Aと同様のユーザースタディ | SUS, 受容率 |

| | 内容 |

## 推薦パターン

### パターン1: 堅実路線（推薦度: ★★★★★）

**候補A（LLM-as-Pricing-Advisor）をメイン + 候補Dの一部（ヘドニック分析）を取り込む**

- 候補Aのアーキテクチャで実装し、価格提案の根拠として候補Dのaspect-WTP推定を組み込む
- 実装スコープが管理可能で、段階的に拡張可能
- LLMエージェント + ABSA拡張 + ユーザースタディという構成で、技術貢献と実務貢献の両方が示せる
- ABSAの拡張は感情5段階化 + 時系列集約 + キーフレーズ強化の3点に絞る

### パターン2: 最大インパクト路線（推薦度: ★★★★）

**候補B（マルチエージェント）をメインに、候補Aの対話機能を統合**

- 学会発表・ジャーナル投稿を狙うなら最もインパクトが大きい
- 実装範囲が大きいため、Agent 1 + Agent 4に集中し、Agent 2, 3は既存手法を利用
- 「AIエージェント × hospitality」は2026年のホットトピック（Dwivedi et al., 2026, 被引用25件）

### パターン3: 方法論的貢献路線（推薦度: ★★★★）

**候補D（因果推論）をメインに、候補Aの対話エージェントを説明層として追加**

- 研究の独自性が最も高い（ギャップ4, 6に直接対応）
- データの制約を乗り越えられるかが鍵
- 計量経済学と機械学習の融合で、学際的な貢献が見込める
